처음으로 SQL Injection 취약점입니다.
지사의 login_proc.php 로그인 처리 코드에서 Prepared Statement를 적용하지 않아 외부 입력값을 동적 문자열로 직접 처리해 악성 쿼리문이 그대로 실행되는 취약점이 발견되었습니다.
기본적으로 sql injection은 값을 계속 바꿔가며 입력해 정보를 탈취하므로 burp suite의 repeater를 사용했고,
공격 방식은 Boolean-Based Blind SQL Injection, ORDER BY 문을 사용해 진행했습니다.
첫번째로 Boolean Blind 방식을 통해 로그인 페이지에서 이와같은 쿼리문을 사용해 database 이름을 탈취할 수 있었고, 다른 db 정보까지도 탈취 가능합니다.
두번째는 ORDER BY 문을 사용한 공격으로 뒤에오는 숫자를 하나씩 올리면서 요청을 보내고 쿼리 실행 실패가 되면 sql 취약점이 존재한다는 뜻이고 에러가 뜨는 숫자의 전 숫자가 컬럼 개수임을 알 수 있습니다. 그리고 컬럼 개수를 알게 되면 이후에 UNION SELECT 공격을 통해 db에 있는 중요한 정보들이 탈취당할 수 있습니다.
이러한 sql injection 취약점을 방어하기위해 기존 취약한 코드에서 prepare 를 사용해 sql 쿼리문을 미리 컴파일해 db에 등록하고, 이후 bindParam 을 사용해 컴파일된 쿼리에 파라미터를 바인딩시켜 공격자가 입력한 값에 명령어가 포함되도 바인딩된 파라미터로 인식하게 만들어 sql injection 방어했습니다.

다음으로 IDOR 취약점입니다.
idor은 서버가 요청자의 권한을 확인하는 인가 단계가 누락되어서 생기는 취약점으로, 지사의 cart_update_proc.php, order_view.php, mypage_update.php 에서 발견되었고, 그 중 cart_update_proc.php을 보겠습니다
burp suite를 이용해 장바구니의 수량을 변경할 때  패킷을 가로채게되면 파라미터가 평문으로 노출된 것을 확인할 수 있고, 간단하게 cart_id와 cart_quantity 를 변경해 타인의 주문내역을 탈취하고 변조까지 가능합니다.
사진을 보시면 단순 화면만 변한게 아니라 실제 데이터베이스 값까지 변한 것을 확인 할 수 있습니다.
이러한 IDOR 취약점을 방어하기위해 prepared statement 를 적용하고 세션 id와 결합 검증을 해 타인의 cart_id로 url을 변조해 전송하더라도 세션값의 userid와 비교해 일치하지 않으면 접근이 차단됩니다.

다음으로는 배포 아키텍쳐입니다
저희는 AWS ec2 1대에 docker compose 기반으로 배포했습니다
docker compose는 web container 와 db container로 이루어져 있고 기본적으로 web으로는 0.0.0.0으로 외부에서 모든 접근을 허용했고 DB는 내부 네트워크에서만 접근 가능하도록 설정했습니다.
docker compose 기반으로 배포한 이유는 컨테이너를 각각 따로 올릴 때 실수가 많이 발생한다고 생각해 docker-compose.yml 파일을 만들어 다중 컨테이너의 명세를 하고, docker-compose up -d 로 동시에 빌드해서 실수없이 안정적으로 구동하고자 했습니다.

마지막으로 트러블 슈팅입니다.
저는 이번 프로젝트에서 모의해킹을 담당하면서 PHP 기반 웹 환경을 직접 구축하고 취약점을 분석하는 역할을 수행했습니다.
이전에 정보보안과 모의해킹 공격에 대해 아무것도 몰랐을 때는 어려운 기술들로 웹사이트를 공격하고 방어하는 것으로 생각했습니다. 
그러나 이번에 직접 취약한 코드를 작성하고 이를 공격하는 과정을 거치면서 느낀점은 공격은 웹사이트의 취약점을 찾아 빈틈을 공격하고 
방어는 기술적으로 많은 것이 필요한 것 보다는 정말 간단한 prepared statement나 세션 검증 같은 시큐어 코드의 한줄이 치명적인 취약점을 방어 할 수 있다는 것을 알 수 있었습니다.

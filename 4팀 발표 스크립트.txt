-- 발표 개요 --
안녕하십니까 유기견 성향 분석 기반 추천 플랫폼 구축 및 정보보호 체계 강화 사업을 맡게된 findeR 입니다.
(인사)
PM을 맡게 된 서서울입니다 프로젝트 발표 시작하겠습니다

목차로는 프로젝트 배경, 분석 및 점검, 개인 발표, Q&A 순으로 발표 진행하도록 하겠습니다.

처음으로 프로젝트 배경입니다.
고객사 딩동독은 점차 늘어나는 사이버 보안 위협에 미리 준비하고 통합 보안 체계 구축을 통해 안정적인 인프라와 운영 환경을 마련하고 사용자의 신뢰성을 확보하고자 이 사업을 추진하게 되었습니다.

딩동독 서비스는 유기견 데이터 기반의 멍! 매칭을 통해 반려견과 입양 가족을 행복하게 만들어주고자 시작된 서비스입니다. 

사업의 규모가 커지면서 회원 정보와 유기견 데이터를 노리는 공격 위험이 늘어났고, 
아직 통합 보안 체계가 잡혀있지 않아 공격 발생 시점과 전체적인 영향 범위를 신속하게 파악하는데 어려움을 겪었습니다.
이로 인해 딩동독 서비스에서 관리하는 멍! 카이브 유기견 데이터 변조 공격 등 서비스의 신뢰성을 위협하는 공격이 늘어날 것을 예상했습니다 .

findeR는 이러한 문제점을 전달받아 종합 보안 체계 구축, 신속한 탐지 및 대응 체계 마련, 보안 위협 선제적 대응, 지속 가능한 보안 운영 기준 수립 크게 4가지로 사업목표를 잡게 되었습니다.

이번 프로젝트를 진행하면서 사용한 도구들입니다.
가상환경에서 미리 테스트하기 위해 vmware와 gns3를 사용하였고, kali linux와 burp suite를 사용해 모의해킹 취약점 점검을 진행하였습니다. 
보안 장비로는 UTM과 SIEM 장비를 사용했고, 구글 드라이브, 노션, 슬랙, drow.io 를 사용해 협업을 진행했습니다.

프로젝트 기간은 총 31일로 크게 프로젝트 관리, 취약점 분석 및 평가, 보안대책 수립 및 조치지원, 모의해킹 문서화 및 보고 파트를 나눠서 진행했습니다.

수행 파트로는 네트워크, 서버, 모의해킹, 보안장비, 배포팀으로 나누어 각 파트당 2명씩 맡이 진행했습니다.

다음으로 분석 및 점검입니다.
저희는 KISA의 2026년 주요정보통신기반시설 기술적 취약점 분석 평가방법 상세가이드를 기준으로 보안성을 점검했고, 이와 같이 사업 제안서를 작성해 이번 프로젝트를 시작했습니다.

본사 자산 목록은 총 33대 중 22대 점검 완료했습니다.
지사 자산 목록은 총 30대 중 21대 점검 완료했습니다.

다음으로 이 그림은 기존 고객사 전체 인프라 구성도입니다.
자세히 보면 이와같이 인프라 취약점으로 보안장비 부재와 보안 설정 미흡, 이중화 부재, 본사와 지사간 사설 통신망의 부재가 있습니다.

다음으로 이러한 취약점을 개선한 전체 인프라 구성도입니다.
개선 사항으로는 DMZ 및 내부방 구축을 통한 외부망과의 분리, L3 Switch 이중화, 보안장비 도입, 본사와 지사간의 VPN 통신이 있습니다.

첫번째로 서비스 연속성을 위한 네트워크 이중화 구조입니다.
코어 스위치가 하나만 존재할 경우 해당 장비 장애시 내부망 전체가 마비되는 단일 장애 지점 리스크가 생기기 때문에 L3 코어 스위치를 두 대 배치하여 이중화를 설정하였습니다.
안정성과 시스코 벤더 최적화를 위해 HSRP 프로토콜을 사용했으며, 루핑을 방지하고 트래픽을 최적화하기 위해 hsrp active경로와 STP 루트 브릿지를 일치시켜 최단거리 라우팅이 가능하도록 하였습니다.

==3p==
실제 l3 스위치의 이중화 설정입니다.
A 스위치의 우선순위를 B 스위치보다 높게 설정하여 액티브 경로를 A 스위치로 고정하였습니다.
또한 양쪽 장비 모두 preempt 설정을 적용하여 장애 복구 시 자동으로 액티브 권한을 회수하여 원래의 최적 경로로 복귀하도록 하였습니다.
이중화 설정 후 A스위치에 장애를 발생시키자 B 스위치가 액티브 권한을 이어받았고, A 스위치가 복구되자 액티브 권한을 다시 회수하는것을 확인하였습니다. 

==4p==
이어서 본사와 지사가 안전하게 통신하기 위한 사설 vpn 통신망 구축입니다.
기업 환경에선 공용 인터넷망을 통해 데이터를 주고받는건 보안상 매우 취약합니다.
그래서 보안을 위해 표준 IPsec VPN을 검토했으나, 저희가 사용한 OSPF 라우팅 프로토콜은 멀티캐스트 트래픽을 사용하기 때문에, 표준 IPsec VPN에서는 전송이 불가능했습니다.
이를 해결하기 위해 본사와 지사의 라우터에 GRE 터널을 설정하여 멀티캐스트 트래픽 전송이 가능하게 만든 후 그 위에 보안 프레임워크인 IPSec설정을 더하여 GRE over IPsec으로 터널을 통과하는 모든 데이터를 보호하도록 설계하였습니다. 

==5p== 
vpn 구축시의 설정은 3단계로 나누어 진행하였습니다.
첫번째 단계는 GRE 가상 터널 생성 과정입니다.양 라우터에 터널 인터페이스를 생성 한 후 본사와 지사의 공인IP를 터널 출발지와 목적지에 매핑하였습니다. 그 후 터널 인터페이스에 ip 주소를 부여하여 본사와 지사간 통신 기반을 만들었습니다.
두번째 단계는 보안 채널 수립 단계입니다. GRE 가상 터널은 자체적인 보안 기능이 없기 때문에 기밀성을 위한 3DES 대칭키 알고리즘과 키 교환을 위한 디피 헬먼 그룹 2 규격을 선언하였습니다. 그리고 프리셰어드 키를 등록하여 본사와 지사 사이의 보안 채널을 수립하였습니다.
세번째 단계는 암호화 적용 과정입니다. 실제 트래픽을 캡슐화해줄 esp 프로토콜을 선언하고 transform set을 생성했습니다. 이후 암호화 정책들과 암호화 대상 트래픽을 지정하는 ACL, 그리고 원격지 피어 정보를 묶은 크립토 맵을 생성하고 터널 인터페이스에 적용함으로써 터널을 통과하는 패킷들이 암호화되도록 설정을 마무리하였습니다. 

==6p==
vpn 설정을 마친 뒤 신뢰성을 검증하기 위한 ping 테스트 결과입니다.
왼쪽을 보시면 본사 개발팀 pc와 지사 DB서버가 서로 ping이 되는것을 알 수 있습니다.
내부망에서 캡쳐한 패킷은 사설 ip정보가 확인되며 icmp 패킷으로 잡히고 있습니다.
반면 외부 공인망에서 캡쳐한 패킷은 라우터의 공인 ip주소가 노출되며 esp 패킷으로 잡히는것을 확인할 수 있습니다.
이를 통해 외부에서 공격자가 패킷을 탈취하더라도 기밀성이 보장된다는것을 확인할 수 있습니다.

==7p==
마지막으로 프로젝트 진행시 발생했던 산출물 형상관리에 대한 트러블슈팅입니다.
산출물 작업시 담당 파트들을 분산 작업한 뒤 메신저나 노션을 통해 공유한 후 취합하는 방식으로 진행했었습니다.
처음에는 파일명 규칙이 버전 형식밖에 없었기 때문에 개인 판단으로 버전을 바꾸는 문제나 버전 혼선으로 일부 작업 내역이 누락된 채 취합되는등의 문제가 생겼습니다. 이후 파일명 규칙에 작성자명과 작성 날짜를 추가하는 조치를 취해 작업을 진행했습니다.
하지만 이 방법 역시 혼동이 생길수 있기 때문에 추후에는 노션등을 활용해 문서 변경 이력을 정리하고, 취합시 검토 및 승인 프로세스를 구축하는등, 체계적인 형상관리 프로세스를 만들어 진행하겠습니다.

서버 파트 개인 발표 시작하겠습니다.

첫 번째로 WAF Server입니다.
Reverse Proxy 아키텍처를 도입하여 외부 사용자가 내부 WEB Server의 IP를 알 수 없도록 은닉하였고,
ModSecurity의 SecPcreMatchLimit 설정을 통해 공격자가 보안 엔진을 마비시키려는 시도를 사전에 차단했습니다.
또한, Referer 검증과 파라미터 규칙 적용을 통해 SQL Injection과 XSS 같은 웹 공격을
L7 계층에서 선제적으로 방어하는 다중 계층 보안을 구축했습니다.

다음은 Geo-match입니다.
MaxMind GeoIP DB를 연동해 사용자의 접속 국가를 실시간으로 하고,
한국(KR) 트래픽만 허용하는 White-list 접근 제어를 구현하여,
서비스 대상이 아닌 국가에서의 불필요한 접근과 공격 접점을 원천 차단했습니다.

다음은 WAF Server의 로그 전송 설정입니다.
rsyslog와 imfile Module을 사용하여 WEB Server의 텍스트 로그를 실시간으로 변환하고,
UDP가 아닌 TCP 전송 방식을 택해 로그 유실을 방지하고 무결성을 보장했습니다.
또한 local6 파실리티로 로그를 체계적으로 분류·태깅하여 관리 효율성을 높였습니다.

두 번째로 WEB Server입니다.
접근 제어로는 UFW 방화벽을 통해 IP Whitelisting 정책을 적용하여 WAF Server로부터 유입되는 Traffic만 허용하도록
정책을 고정해 서비스 노출 면적을 최소화하였고, System Hardening으로는 cups와 avahi-daemon을 비활성화하여
악성코드가 네트워크 내에서 서버 간에 이동하는 측면 이동 경로를 물리적으로 차단했습니다.

세 번째로 WAS Server입니다.
connectionTimeout과 maxParameterCount 제한으로 DoS 공격으로부터 자원을 보호하고,
쿠키에 HTTPOnly와 Secure 속성을 적용해 Session Hijacking과 스니핑을 방지했습니다.
또한, server 속성을 은닉하고 에러 페이지를 커스텀 리다이렉트 하여 서버 정보가 노출되는 경로를 완벽히 차단하였습니다.

네 번째로 DNS Server입니다.
DNSSEC을 도입해 디지털 서명(RRSIG)과 공개 키(DNSKEY)로 도메인 정보의 위·변조를 검증하여 DNS Spoofing을 차단했습니다.
추가로 Locking Percent 100% 설정과 LLMNR Protocol 비활성화를 통해,
Cache Poisoning 공격과 내부망 인증 정보 탈취 경로를 차단했습니다.

마지막으로 LOG Server입니다.
전체 로그 데이터 흐름은 WAF Server에서 시작된 이벤트가 LOG Server와 Aggregator를 거쳐 SIEM까지 전송되는 구조입니다.
앞서 설정한 rsyslog와 imfile 모듈을 통해 모든 보안 이벤트가 실시간으로 수집되고 있습니다.
특히 Tagging을 통해 장비별로 로그를 체계적으로 분류함으로써,
공격자가 원본 서버의 로그를 인멸하더라도 SIEM에 기록된 타임라인을 통해 공격의
시작점부터 경로까지 정확히 추적할 수 있는 관제 환경을 구축하였습니다.

마지막으로, 이번 프로젝트를 수행하며 경험한 트러블 슈팅과 향후 보안 계획에 대해 말씀드리겠습니다.
보안 시스템 구축 시 가장 중요한 것은 서비스의 안정성이라고 생각해 초기에는 강력한 보안 규칙인 OWASP_CRS를 바로 적용하기보다,
서비스의 정상 트래픽 패턴을 분석하고 가시성을 확보하는 데 집중했습니다.
이 과정에서 로그 파일의 권한 설정과 대량 로그 전송 시 발생하는 지연 이슈가 있었으나,
권한 관리 체계화를 통해 안정적인 관제 환경을 구현할 수 있었습니다.
이렇게 확보된 가시성을 바탕으로, 향후에는 OWASP_CRS를 단계적으로 도입하여 보안 규칙을 정교화하고,
iptables를 활용해 인프라 네트워크 격리 수준을 더욱 고도화할 예정입니다.

이상으로 개인 발표를 마치겠습니다.

UTM ACL 정책 (50초)
먼저 UTM에서는 조직 환경에 맞는 접근 제어 정책을 구성했습니다.
단순히 모든 접근을 허용하는 것이 아니라, 부서별 업무에 필요한 통신만 허용하도록 정책을 설정했습니다.
또한 웹, DNS, PHP, DB, 로그 서버처럼 서버 역할에 따라 접근 권한을 구분하여 최소 권한 원칙으로 정책을 구성했습니다.
이를 통해 불필요한 접근은 차단하면서도 업무에 필요한 통신은 정상적으로 이루어지도록 구성했습니다.

DoS 및 포트스캔 방어 (50초)
다음은 UTM의 보안 기능을 활용한 공격 방어 설정입니다.
먼저 포트 스캔 방지 기능을 적용하여 비정상적인 포트 탐색 시도를 차단했습니다.
또한 UDP Flood와 TCP Flood 방어 기능을 활성화하여 대량의 트래픽을 이용한 DoS 공격에도 대응할 수 있도록 구성했습니다.

SIEM 1장 - 로그 수집 구성
다음은 SIEM 구성입니다.
WAF에서 탐지된 보안 이벤트를 SIEM으로 연동하여 하나의 화면에서 확인할 수 있도록 구성했습니다.
이를 통해 여러 장비의 로그를 개별적으로 확인하는 것이 아니라 중앙에서 통합 관리할 수 있는 환경을 구축했습니다.

SIEM 2장 - JSON 변환 및 정규식
WAF에서 생성되는 원본 로그는 그대로는 분석하기 어렵기 때문에 JSON 형태로 변환한 뒤 SIEM으로 전달했습니다.
이후 SIEM에서는 정규식을 이용하여 공격 시간, 공격 IP, 탐지 룰, 공격 유형 등의 정보를 각각의 필드로 추출하도록 설정했습니다.
이를 통해 필요한 정보를 보다 쉽게 검색하고 분석할 수 있도록 구성했습니다.

SIEM 3장 - 탐지 및 정책
마지막으로 수집된 로그를 기반으로 탐지 정책을 적용했습니다.
SQL Injection이나 XSS, Command Injection 등 주요 웹 공격이 발생하면 이벤트가 생성되도록 설정했으며, 공격 유형별로 로그를 구분하여 확인할 수 있도록 구성했습니다.
이러한 구성을 통해 공격 발생 여부를 빠르게 파악하고 효율적으로 대응할 수 있는 환경을 구축했습니다.

트러블슈팅 문제 발생 (1분)
이어서 네트워크 구성 과정에서 겪었던 트러블슈팅 입니다
프로젝트를 진행하면서 가장 어려웠던 문제는 ACL 적용 이후 발생한 통신 장애였습니다.
ACL을 적용한 이후 DHCP가 정상적으로 동작하지 않았고, HSRP 상태에도 문제가 발생하면서 부서 간 통신이 모두 끊기는 현상이 발생했습니다.
원인을 확인해 보니 업무 통신만 고려하여 ACL을 작성하면서 마지막에 적용한 deny ip any any 정책 때문에 DHCP와 HSRP Hello, ICMP 같은 운영에 필요한 패킷까지 함께 차단되고 있었습니다.
즉, 보안 정책은 올바르게 작성했다고 생각했지만 네트워크 운영에 필요한 트래픽까지 막고 있었던 것이 원인이었습니다.

해결 과정 (1분)
문제를 해결하기 위해 ACL 상단에 DHCP, HSRP Hello, ICMP 등 운영에 반드시 필요한 트래픽을 먼저 허용하도록 정책을 수정했습니다.
그 이후 부서 간 통신 정책과 차단 정책을 순서대로 적용하여 ACL을 다시 구성했습니다.
수정 후에는 DHCP가 정상적으로 주소를 할당했고, HSRP 상태도 정상적으로 유지되었으며 부서 간 통신 역시 모두 정상적으로 동작하는 것을 확인했습니다.
이번 경험을 통해 보안 정책은 단순히 차단하는 것이 아니라, 서비스 운영에 필요한 트래픽까지 함께 고려해야 안정적인 보안 구성이 가능하다는 점을 배울 수 있었습니다.

이번 실습에서는 개인정보 수정 기능을 대상으로 공격을 수행했습니다 
공격자는 악성 페이지에 개인정보 수정 요청을 숨겨, 사용자가 해당페이지에 접속하는 것만으로 의도치 않은 정보 변경이 이루어지도록 구성했습니다  
다음은 실제 분석 및 공격 결과입니다 

슬라이드 2
사용자가 악성페이지에 접속하면 브라우저는 로그인 상태를 유지하기 위해 저장되어 있던  세션 쿠키를 자동으로 함께 전송합니다. 
브라우저는 요청이 정상페이지 에서 만들어졌는지 판단하지 않고, 로그인에 필요한 쿠키만 있으면 자동으로 포함합니다 
서버 역시 세션이 유효한지만 확인했고, 요청이 어디에서 생성되었는지 확인하는 Origin 검증이나 CSRF Token 검증이 존재하지 않아 
서버는 공격자가 만든 요청을 정상 사용자의 요청으로 처리했고, 화면에서 보시는 것처럼 사용자의 이메일주소나, 닉네임이 변경되는 것을 확인 
할 수 있습니다 
즉, 사용자가 직접 정보를 수정하지 않았음에도 자신의 권한이 공격에 이용된 사례입니다 
이를 방어하기 위해 세 가지 보안 기능을 적용했습니다.

슬라이드 3
첫째는 CSRF Token입니다.
서버가 사용자마다 임의의 토큰을 생성하고 요청마다 함께 전달하도록 했습니다. 서버는 전달받은 토큰이 저장된 값과 일치하는 경우에만 요청을 처리하기 때문에 공격자가 임의로 생성한 요청은 거부됩니다.
둘째는 Origin 검증입니다.
HTTP Origin 헤더를 확인하여 요청이 허용된 도메인에서 생성되었는지 검사하도록 했습니다. 허용되지 않은 사이트에서 발생한 요청은 서버에서 차단하도록 구현했습니다.
셋째는 SameSite Cookie 설정입니다.
브라우저가 다른 사이트에서 발생한 요청에는 인증 쿠키를 자동으로 전송하지 않도록 하여 CSRF 공격이 성립하기 어려운 환경을 구성했습니다.

슬라이드 4 
다음은 파일 업로드 취약점입니다.
진단 대상은 상품 등록 처리 모듈인 product_add_proc.php였습니다.
분석 결과 서버에서는 업로드되는 파일의 확장자와 실제 파일 내용을 검증하는 로직이 존재하지 않았습니다.
원래는 이미지 파일만 등록되어야 하는 기능이지만 이러한 검증이 없으면 공격자는 이미지 대신 악성 PHP 파일을 업로드할 수 있습니다.
즉, 서버가 업로드되는 파일을 그대로 신뢰하는 상태였습니다.

슬라이드 5
실제 공격 과정입니다.
공격자는 정상 상품 등록 기능을 이용하여 shell.php 파일을 업로드했습니다.
이때 서버는 허용된 확장자인지 확인하는 필터링이 없었고, 클라이언트가 전송한 Content-Type 헤더도 검증하지 않았습니다.
하지만 Content-Type 헤더는 사용자가 얼마든지 변경할 수 있는 값이므로 서버가 그대로 신뢰해서는 안 됩니다.
결국 서버는 악성 PHP 파일을 정상 파일로 인식했고, 응답 패킷에서도 HTTP 200 OK를 반환하며 업로드를 허용한 것을 확인할 수 있었습니다.

슬라이드 6
업로드 이후 공격 과정입니다.
공격자는 개발자 도구를 이용해 업로드된 파일의 저장 위치를 확인했습니다.
이후 해당 웹쉘을 호출하면서 cmd 파라미터를 전달하자 서버에서 운영체제 명령이 실행되었습니다.
화면에서 보시는 것처럼 /etc/passwd 파일 내용이 출력되었고, 이는 서버에서 임의의 명령을 실행할 수 있다는 것을 의미합니다.
이러한 취약점은 단순한 정보 조회에 그치지 않고, 추가 악성 프로그램 설치나 서버 장악으로도 이어질 수 있는 위험한 취약점입니다.

슬라이드 7
이를 방어하기 위해 세 가지 보안 대책을 적용했습니다.
첫째, 화이트리스트 방식을 적용하여 허용된 이미지 확장자만 업로드할 수 있도록 했습니다.
둘째, 클라이언트가 보내는 Content-Type 헤더는 신뢰하지 않고, 서버가 파일 내용을 직접 검사하여 MIME 타입을 검증하도록 구현했습니다.
이 과정에서 파일의 Magic Number까지 확인하기 때문에 확장자만 변경한 위장 파일은 업로드할 수 없습니다.
마지막으로 파일명을 난수 형태로 저장하여 공격자가 업로드 경로를 예측하기 어렵도록 했습니다.
이를 통해 파일에 직접 접근하려는 시도까지 함께 줄일 수 있도록 구성했습니다.
이어서 웹쉘 공격 시연영상을 시청하시겠습니다 

마지막으로 CSRF 보안설정을 하면서 겪었던 트러블 슈팅에 대해 설명하겠습니다 
SameSite 쿠키를 적용하기 위해 PHP의 session_set_cookie_params()를 사용했지만, 개발자 도구에서 확인한 결과 SameSite=Lax 속성이 적용되지 않는 문제가 발생했습니다.
원인을 분석해 보니 사용 중인 PHP 7.2 버전에서는 배열 형태의 SameSite 설정을 지원하지 않아 해당 코드가 무시되고 있었습니다.
그래서 애플리케이션 코드를 계속 수정하는 대신, Apache의 Header always edit 기능을 사용해 모든 응답의 Set-Cookie 헤더를 강제로 재작성하도록 설정했습니다.
이를 통해 모든 응답에서 SameSite=Lax 속성이 정상적으로 적용되는 것을 확인할 수 있었습니다.
이상으로 개인 발표를 마치고, 이어서 팀장 서서울 님의 발표가 있겠습니다. 감사합니다

처음으로 SQL Injection 취약점입니다.
지사의 login_proc.php 로그인 처리 코드에서 Prepared Statement를 적용하지 않아 외부 입력값을 동적 문자열로 직접 처리해 악성 쿼리문이 그대로 실행되 로그인 우회에 성공하는 취약점이 발견되었습니다.
기본적으로 sql injection은 값을 계속 바꿔가며 입력해 정보를 탈취하므로 burp suite의 repeater를 사용했고,
공격 방식은 Boolean-Based Blind SQL Injection, ORDER BY 문을 사용해 진행했습니다.
첫번째로 Boolean Blind 방식을 통해 로그인 페이지에서 이와같은 쿼리문을 사용해 database 이름을 탈취할 수 있었고, 다른 db 정보까지도 탈취 가능했습니다.
두번째는 ORDER BY 문을 사용한 공격으로 뒤에오는 숫자를 하나씩 올리면서 요청을 보내고 쿼리 실행 실패가 되면 sql 취약점이 존재한다는 뜻이고 에러가 뜨는 숫자의 전 숫자가 컬럼 개수임을 알 수 있습니다.
그리고 컬럼 개수를 알게 되면 이후에 UNION SELECT 공격을 통해 db에 있는 중요한 정보들이 탈취당할 수 있습니다.
이러한 sql injection 취약점을 방어하기위해 기존 취약한 코드에서 prepare 를 사용해 sql 쿼리문을 미리 컴파일해 db에 등록하고, 
이후 bindParam 을 사용해 컴파일된 쿼리에 파라미터를 바인딩시켜 공격자가 입력한 값에 명령어가 포함되도 바인딩된 파라미터로 인식하게 만들어 sql injection을 방어했습니다.

다음으로 IDOR 취약점입니다.
idor은 서버가 요청자의 권한을 확인하는 인가 단계가 누락되어서 생기는 취약점으로, 지사의 cart_update_proc.php, order_view.php, mypage_update.php 에서 발견되었고, 그 중 cart_update_proc.php을 보겠습니다
burp suite를 이용해 장바구니의 수량을 변경할 때  패킷을 가로채게되면 파라미터가 평문으로 노출된 것을 확인할 수 있고, 간단하게 cart_id와 cart_quantity 를 변경해 타인의 주문내역을 탈취하고 변조까지 가능했습니다.
사진을 보시면 단순 화면만 변한게 아니라 실제 데이터베이스 값까지 변한 것을 확인 할 수 있습니다.
이러한 IDOR 취약점을 방어하기위해 prepared statement 를 적용해 1차적으로 파리미터가 변조되는 것을 막고, 세션 id와 cart_id를 결합 검증을 해 타인의 cart_id로 url을 변조해 전송하더라도 세션값의 user_id와 비교해 일치하지 않으면 접근이 차단됩니다.

다음으로는 배포 아키텍쳐입니다
저희는 AWS ec2 1대를 기반으로, 그 위에 docker compose 를 통해 web container와 db container를 구동했습니다.
기본적으로 외부에서는 80/443 포트를 통해 web 서비스에만 접근 가능하도록 하고, db는 외부와 격리된 내부 네트워크에서만 통신하도록 Security Group을 설정해 보안을 강화했습니다.  그리고 운영 안정성을 위해 컨테이너가 재시작되어도 데이터가 유실되지 않도록 volume을 사용하여 영속성을 확보했습니다.

docker compose를 활용해 배포를 진행한 이유는 크게 두 가지입니다.
첫째, 휴먼 에러 최소화입니다. 컨테이너를 개별적으로 빌드하고 실행하다 보면 설정 누락이나 버전 불일치 같은 실수가 발생하기 쉽습니다.

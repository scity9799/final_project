목차로는 프로젝트 배경, 분석 및 점검, 개인 발표, Q&A 순으로 발표 진행하도록 하겠습니다.

처음으로 프로젝트 배경입니다.
고객사 딩동독은 점차 늘어나는 사이버 보안 위협에 미리 준비하고 통합 보안 체계 구축을 통해 안정적인 인프라와 운영 환경을 마련하고 사용자의 신뢰성을 확보하고자 이 사업을 추진하게 되었습니다.

딩동독 서비스는 유기견 데이터 기반의 멍! 매칭을 통해 반려견과 입양 가족을 행복하게 만들어주고자 시작된 서비스입니다. 사업의 규모가 커지면서 회원 정보와 유기견 데이터를 노리는 공격 위험이 늘어났고, 아직 통합 보안 체계가 잡혀있지 않아 딩동독 서비스에서 관리하는 멍! 카이브 유기견 데이터 변조 공격 등 서비스의 신뢰성을 위협하는 공격이 늘어날 것을 예상했습니다 .

findeR는 이러한 문제점을 전달받아 종합 보안 체계 구축, 신속한 탐지 및 대응 체계 마련, 보안 위협 선제적 대응, 지속 가능한 보안 운영 기준 수립 크게 4가지로 사업목표를 잡게 되었습니다.

수행 파트로는 네트워크, 서버, 모의해킹, 보안장비, 배포팀으로 나누어 각 파트당 2명씩 맡이 진행했습니다.

처음으로 SQL Injection 취약점입니다.
지사의 login_proc.php 로그인 처리 코드에서 Prepared Statement를 적용하지 않아 외부 입력값을 동적 문자열로 직접 처리해 악성 쿼리문이 그대로 실행되는 취약점이 발견되었습니다.
기본적으로 sql injection은 값을 계속 바꿔가며 입력해 정보를 탈취하므로 burp suite의 repeater를 사용했고,
공격 방식은 Boolean-Based Blind SQL Injection, ORDER BY 문을 사용해 진행했습니다.
첫번째로 Boolean Blind 방식을 통해 로그인 페이지에서 이와같은 쿼리문을 사용해 database 이름을 탈취할 수 있었고, 다른 db 정보까지도 탈취 가능합니다.
두번째는 ORDER BY 문을 사용한 공격으로 뒤에오는 숫자를 하나씩 올리면서 요청을 보내고 쿼리 실행 실패가 되면 sql 취약점이 존재한다는 뜻이고 에러가 뜨는 숫자의 전 숫자가 컬럼 개수임을 알 수 있습니다. 그리고 컬럼 개수를 알게 되면 이후에 UNION SELECT 공격을 통해 db에 있는 중요한 정보들이 탈취당할 수 있습니다.
이러한 sql injection 취약점을 방어하기위해 기존 취약한 코드에서 prepare 를 사용해 sql 쿼리문을 미리 컴파일해 db에 등록하고, 이후 bindParam 을 사용해 컴파일된 쿼에 파라미터를 바인딩시켜 공격자가 입력한 값에 명령어가 포함되도 바인딩된 파라미터로 인식하게 만들어 sql injection 방어했습니다.

다음으로 IDOR 취약점입니다.
idor은 서버가 요청자의 권한을 확인하지않아서 생기는 취약점으로, 지사의 cart_update_proc.php, order_view.php, mypage_update.php 에서 발견되었고, 그 중 cart_update_proc.php을 보겠습니다
burp suite를 이용해 장바구니의 수량을 변경할 때  패킷을 가로챈 뒤 간단하게 cart_id와 cart_quantity 를 변경해 타인의 주문내역을 탈취하고 변조까지 가능합니다.
사진을 보시면 단순 화면만 변한게 아니라 실제 데이터베이스 값까지 변한 것을 확인 할 수 있습니다.
이러한 IDOR 취약점을 방어하기위해 prepared statement 를 적용하고 세션 id와 결합 검증을 해 타인의 cart_id로 url을 변조해 전송하더라도 세션값의 userid와 비교해 일치하지 않으면 접근이 차단됩니다.

다음으로는 배포 아키텍쳐입니다
저희는 AWS ec2 1대에 docker compose 기반으로 배포했습니다
docker compose는 web container 와 db container로 이루어져 있고 기본적으로 web으로는 0.0.0.0으로 외부에서 모든 접근을 허용했고 DB는 내부 네트워크에서만 접근 가능하도록 설정했습니다.
docker compose 기반으로 배포한 이유는 컨테이너를 각각 따로 올릴 때 실수가 많이 발생한다고 생각해 docker-compose.yml 파일을 만들어 다중 컨테이너의 명세를 하고, docker-compose up -d 로 동시에 빌드해서 실수없이 안정적으로 구동하고자 했습니다.

마지막으로 트러블 슈팅입니다.
저는 이번 프로젝트에서 모의해킹을 담당하면서 php 웹 사이트도 만들어보면서 취약한 코드와 시큐어 코드를 보게 되었습니다.
이전에 모의해킹 공격에 대해 아무것도 몰랐을 때는 엄청나게 어려운 것으로 생각했으나 실상은 정말 간단한 코드의 추가만으로도 1차적으로 방어가 가능하다는 것을 알게되었고 개발을 하거나 취약점 분석을 할 때 항상 먼저 생각하면서 확인해야겠다고 생각했습니다.

-- 윤철민
안녕하십니까 PL을 맡게 된 윤철민입니다
서버 파트 개인발표 시작하겠습니다.


1. WAF Server(1 ~ 3 p)
먼저, Reverse Proxy 아키텍처 및 L7 방어를 통한 인프라 구현입니다.
WAF는 웹 서버 전단에서 모든 트래픽을 선제적으로 검역하는 첫 번째 보안 관문입니다.
특히 L3/L4 단계에서 탐지하지 못하는 SQLi, XSS 등 애플리케이션 계층의 고도화된 위협을 효과적으로 차단합니다.
또한, Geo-blocking을 통한 공격 표면을 축소했습니다. MaxMind GeoIP DB를 사용하여 서비스 대상이 아닌 국가의 트래픽을 원천 차단함으로써,
공격 가능한 접점을 최소화하는 정밀한 접근 제어를 구현했습니다.
마지막으로 통합 로그 파이프라인입니다. rsyslog의 imfile 모듈을 사용하여 텍스트 로그를 실시간으로 변환하고,
TCP 전송 방식을 통해 데이터 유실 없는 무결성 높은 로그 전송 체계를 갖췄습니다.

2. WEB & DNS Server(4 ~5p)
다음으로 내부 서버들의 보안 강화입니다.
웹 서버는 IP Whitelisting을 통해 WAF 서버에서 유입되는 패킷만 수신하도록 고정하여,
공격자가 웹 서버에 직접 접근하는 경로를 차단했습니다.
DNS 서버는 DNSSEC를 적용하여 응답에 디지털 서명을 포함했습니다. 이를 통해 DNS Spoofing이나 Cache Poisoning 공격으로부터
도메인 정보의 위·변조를 원천 봉쇄했습니다. 양쪽 모두 불필요한 데몬과 포트를 비활성화하는 시스템 하드닝을 완료했습니다.

3. LOG Server(6p)
로그 서버는 저희 인프라의 '블랙박스'입니다. 모든 WAF 및 시스템 로그를 실시간으로 중앙화하여,
공격자가 원본 서버에서 로그를 인멸하더라도 외부 서버에 흔적이 남도록 함으로써
추적 가능성(Accountability)을 완벽하게 확보했습니다.

4. 트러블 슈팅(7p)
첫째, 로그가 수신되지 않는 경우입니다. netstat -antp | grep 514 명령어로 로그 서버가 포트를 정상적으로 리스닝 중인지,
방화벽(iptables/firewalld)에서 514 포트 통신이 양 서버 간에 허용되어 있는지 확인해야 합니다.

둘째, 로그는 들어오지만 내용이 비어있는 경우입니다. 이는 대부분 권한 문제입니다.
rsyslog 프로세스가 해당 로그 파일에 접근할 권한이 없는 경우이므로, ls -l로 권한을 확인한 뒤
rsyslog 유저에게 읽기 권한을 부여하는 조치를 취합니다.

==1p==
안녕하십니까 팀원 오명근입니다.
(인사)
네트워크 파트 개인발표를 시작하겠습니다.

==2p==
첫번째로 서비스 연속성을 보장하기 위한 네트워크 이중화 구조입니다.
코어 스위치가 하나만 존재할 경우 해당 장비에 장애가 발생하면 내부 전체망이 마비되는 단일 장애 지점 리스크가 생기기 때문에 저희는 L3 코어 스위치를 두 대 배치하여 이중화를 설정하였습니다.
안정성과 시스코 벤더 최적화를 위해 HSRP 프로토콜을 사용했으며, 루핑을 방지하고 트래픽을 최적화하기 위해 hsrp active경로와 STP 루트 브릿지를 일치시켜 최단거리 라우팅이 가능하도록 하였습니다.

==3p==
실제 l3 스위치의 이중화 설정입니다.
보시는것처럼 주 장비 역할을 하는 코어 L3 A 스위치의 우선순위를 110으로, 백업 장비인 코어 L3 B 스위치의 우선순위를 100으로 설정하여 액티브 경로를 A 스위치로 고정하였습니다.
또한 양쪽 장비 모두 preempt 설정을 적용하여 장애 복구 시 관리자의 개입 없이도 자동으로 액티브 권한을 회수하여 원래의 최적 경로로 복귀하도록 하였습니다.
이중화 설정 후 A스위치의 전원을 끄자 B 스위치가 액티브 권한을 이어받아 패킷을 처리하였고, A 스위치가 복구되자 액티브 권한을 다시 회수하는것을 확인하였습니다. 

==4p==
이어서 본사와 지사가 안전하게 통신하기 위한 사설 vpn 통신망 구축입니다.
기업 환경에선 원격지 간에 공용 인터넷망을 통해 그대로 데이터를 주고받는건 보안상 매우 취약합니다.
또한 저희가 사용한 OSPF 라우팅 프로토콜은 멀티캐스트 트래픽을 사용하여, 표준 IPsec VPN에서는 전송이 불가능했습니다.
이를 해결하기 위해 본사와 지사의 라우터에 GRE 터널을 설정하여 OSPF 동적 라우팅을 위한 멀티캐스트 트래픽 전송이 가능하도록 하였습니다.
하지만 GRE 자체에는 암호화가 없기 때문에 그 위에 보안 프레임워크인 IPSec설정을 더한 GRE over IPsec으로 터널을 통과하는 모든 데이터를 보호하도록 설계하였습니다. 

==5p== 
vpn 구축시의 설정은 3단계로 나누어 진행하였습니다.
첫번째 단계는 GRE 가상 터널 생성 과정입니다.양 라우터에 터널 인터페이스를 생성 한 후 본사와 지사의 공인IP를 터널 출발지와 목적지에 매핑하였습니다. 그 후 터널 내부에서 사용할 ip 주소를 부여하여 본사와 지사간 통신이 가능하게 만들었습니다.
두번째 단계는 보안 채널 수립 단계입니다. GRE 가상 터널은 자체적인 보안 기능이 없기 때문에 ISAKMP 정책을 설정하여 기밀성을 위한 3DES대칭키 알고리즘과 키 교환을 위한 디피 헬먼 그룹 2 규격을 선언하였습니다. 그리고 프리셰어드 키를 등록하여 본사와 지사 사이의 보안 채널을 수립하였습니다.
세번째 단계는 암호화 적용 과정입니다. 실제 트래픽을 캡슐화해줄 esp 프로토콜을 선언하고 transform set을 생성했습니다. 이후 암호화 정책들과 암호화 대상 트래픽을 지정하는 ACL, 그리고 원격지 피어 정보를 묶은 크립토 맵을 생성하고 터널 인터페이스에 적용함으로써 터널을 통과하는 패킷들이 자동으로 암호화되도록 설정을 마무리하였습니다.

트러블슈팅 문제 발생 (1분)

이어서 네트워크 구성 과정에서 겪었던 트러블슈팅 입니다
프로젝트를 진행하면서 가장 어려웠던 문제는 ACL 적용 이후 발생한 통신 장애였습니다.
ACL을 적용한 이후 DHCP가 정상적으로 동작하지 않았고, HSRP 상태에도 문제가 발생하면서 부서 간 통신이 모두 끊기는 현상이 발생했습니다.
원인을 확인해 보니 업무 통신만 고려하여 ACL을 작성하면서 마지막에 적용한 deny ip any any 정책 때문에 DHCP와 HSRP Hello, ICMP 같은 운영에 필요한 패킷까지 함께 차단되고 있었습니다.

해결 과정 (1분)

문제를 해결하기 위해 ACL 상단에 DHCP, HSRP Hello, ICMP 등 운영에 반드시 필요한 트래픽을 먼저 허용하도록 정책을 수정했습니다.
그 이후 부서 간 통신 정책과 차단 정책을 순서대로 적용하여 ACL을 다시 구성했습니다.
수정 후에는 DHCP가 정상적으로 주소를 할당했고, HSRP 상태도 정상적으로 유지되었으며 부서 간 통신 역시 모두 정상적으로 동작하는 것을 확인했습니다.
이번 경험을 통해 보안 정책은 단순히 차단하는 것이 아니라, 서비스 운영에 필요한 트래픽까지 함께 고려해야 안정적인 보안 구성이 가능하다는 점을 배울 수 있었습니다.
